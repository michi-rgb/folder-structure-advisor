# folder_advisor v2 設計（ゼロから再構築）

## 0. 解決する課題

職場で必要な情報（資料・アプリ）に短時間でアクセスできず、信頼性も判断できない。

| # | 原因 | 本ツールでの対応 |
|---|---|---|
| 1 | 格納・命名・版管理のルールがない | 格納原則・標準フォルダ体系・命名規則・版管理ルールを提案（正式版/作業中を場所で分離） |
| 2 | チーム/サイト作成が無制限で乱立 | 作成統制（申請制・オーナー登録必須・棚卸し）をガバナンス提案に含める |
| 3 | 「正」のオーナー未定義 | 体系案の各フォルダにオーナー役割を割り当てて提案 |
| 4 | 廃棄/アーカイブ運用の不在 | 保持・アーカイブ・廃棄ルールを提案。長期未更新フォルダを棚卸しリスト化 |
| 5 | プラットフォーム構成の重複 | 「どこに何を置くか」の格納原則を提案（複数ソースをスキャンして統合検討の材料に） |
| 6 | 正・最新版を識別できない | 更新日時を低信頼として扱い、ファイル名の版番号を正とする命名・版管理を提案。版乱立・混在を機械検出 |

## 1. 旧版（v1）からの方針転換

v1 はファイル単位の重複統合・最小移動が主軸で、次の問題があった。

- ファイル内容のハッシュ計算が必要 → OneDrive のクラウド専用ファイルを全量ダウンロードしてしまい通信量が莫大
- 提案がファイル移動の羅列で、「体系・ルールの再設計」という本来の課題に届かない
- API キー前提の認証で、キー配布が社内ポリシーに合わない

v2 の柱:

1. **フォルダ体系（構造）を主対象**にする。個々のファイルではなく、フォルダ単位の統計と課題シグナルを収集する
2. **メタデータのみ**（内容を読まない・送らない）で通信量とリスクを最小化する
3. **LLM を提案の主エンジン**にし、体系案 + ルール文書 + 移行計画まで構造化 JSON で生成させる
4. 認証は **Azure CLI（Entra ID）** に統一し、API キーを不要にする

## 2. 全体フロー

```
scan（体系取得: ローカル / OneDrive・SharePoint 同期フォルダ）
  → analyze（ルール所見: 版乱立・混在・散在・平置き・深層・未更新・汎用名）
  → digest（フォルダ単位1行サマリへ圧縮。上限400フォルダ）
  → LLM（Azure OpenAI, 1回呼び出し, JSON応答）… 失敗時はルールベースへフォールバック
  → report（report.html / 運用ルール案.md / move_plan.csv）
```

## 3. データ取得と通信量削減

### 3.1 ローカル / OneDrive 同期フォルダ（`scan_local.py`）
- `os.scandir` の stat 情報のみ使用。ファイルは open しない
- OneDrive「ファイルオンデマンド」のプレースホルダは内容アクセスしない限り
  ダウンロードされない → **走査による通信はゼロ**
- Windows のファイル属性（`RECALL_ON_DATA_ACCESS` 等）でクラウド専用ファイルを
  識別・カウントし、レポートで「ダウンロードせず走査した」ことを可視化

### 3.2 Graph API を使わない理由
Microsoft Graph API の利用は職場で制限されているため採用しない。
OneDrive / SharePoint は「同期」機能で作られる同期済みローカルフォルダを
3.1 と同じ経路で走査する。メタデータしか読まないため、ファイルオンデマンドの
実体ダウンロードは発生せず、クラウド API 呼び出しもゼロ。
外部通信は Azure OpenAI への提案依頼（1 回）のみとなる。

### 3.3 収集する情報（フォルダ単位のみ）
件数・サブフォルダ数・容量・拡張子内訳・最終更新月（低信頼の参考値）・
代表ファイル名（最大 8 件）・課題シグナル（版番号/作業中/確定トークン数、
同一系列の最大並存数）。**ファイル一覧は保存しない。**

## 4. LLM 連携（Azure OpenAI / Azure CLI 認証）

- `azure-identity` の `AzureCliCredential` + `get_bearer_token_provider`
  （スコープ `https://cognitiveservices.azure.com/.default`）を
  `AzureOpenAI(azure_ad_token_provider=...)` に渡す。API キー不要・トークン自動更新
- 必要ロール: **Cognitive Services OpenAI User**（リソースに対して）
- 呼び出しは提案生成につき **1 回**。`response_format=json_object` で
  スキーマ（principles / target_tree / folder_mapping / naming_rules /
  versioning_rules / governance / lifecycle / quick_wins）を強制
- プロンプトには課題定義（上表の原因 1〜6）とルール所見を埋め込み、
  提案の根拠を観測された事実に固定する
- 設定不足・認証失敗・API エラーはすべて捕捉し、テンプレート提案に
  フォールバック（ツールとしては常に完走）

## 5. 出力

- `report.html` — 自己完結（外部 CDN・外部リソースなし。テストで担保）。
  サマリカード／課題所見表／改善後体系表／before・after 折り畳みツリー／
  移行計画表／各種ルール
- `運用ルール案.md` — そのまま回覧・改訂できるルール文書ドラフト
- `move_plan.csv` — フォルダ移行計画（UTF-8 BOM、Excel で文字化けしない）
- `scan.json` — スキーマ版付き。`propose --scan` で再利用可能

## 6. 明示的な非対象

- 実ファイルの移動・削除・リネーム（提案のみ。移行は人が合意して実施）
- ファイル内容の解析・全文検索・重複ハッシュ（v1 で通信量問題の原因）
- SharePoint の 2 系統統合そのものの実行（判断材料と原則の提案まで）
